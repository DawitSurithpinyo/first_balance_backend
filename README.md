## Detailed tech stack
- Python with Flask.
- MongoDB for the main user data.
- Redis for in-memory data on server-side sessions and rate limiting. Considering to use it for message queue too in the future.
- Pydantic for validating request body, data from DB, and response.
- Bandit SAST, used in both local development and GitHub Actions CI/CD
- CodeQL SAST in GitHub Actions CI/CD

## Brief functionalities
NOTE: Currently, the front end is being remade / refactored. This is the old front end.

### Dashboard page
- View how much spent in total, how much earned in total, and remaining balance.
- More features to add in the future

## API endpoints

API host: <u>https://\<domain>/api</u>

Main response body:
```json
{
    "success": boolean,
    "message": string,
    "messageCode": string to encode specific error or success case,
    "data": any + may be omitted,
    "timestamp": string as ISO 8601 UTC datetime 
}
```

As soon as the user lands on the landing page, they will receive a CSRF token. Client must include the token as a header with `X-CSRF-Token` as key in non-GET requests.

### Auth / user endpoints

#### POST /auth/googleLogin
<u>Endpoint</u>: [API host]/auth/googleLogin

<u>Description</u>: Take Google OAuth code from front end, validate, generate a server-side session and CSRF token, and return the user profile from Google.

<u>Rate limited</u>: No

<u>Request header</u>
| Name | Note |
| ---- | ---- |
| X-CSRF-Token| CSRF token generated by server |

<u>Request body</u>
| Name | Data type | Optional | Note
| ---- | --------- | -------- | ----

<u>Response HTTP status code</u>
| Status | Description |
| ------ | ----------- |
| 200 | OK |
| 400 | Bad request |
| 401 | Unauthorized |
| 500 | Internal error |

^ 401 is for invalid CSRF token of pre-login session

<u>Response body</u>
| Name | Data type | Note |
| ---- | --------- | ---- |
| userID | string | |
| userEmail | string | |
| userName | string | |
| signUpChoice | literal string 'GOOGLE' | |

<u>Response header</u>
| Name | Note |
| ---- | ---- |
| X-CSRF-Token| CSRF token generated by server |

#### GET /auth/getCredentials
<u>Endpoint</u>: [API host]/auth/getCredentials

<u>Description</u>: Get user profile. This is called on refresh and landing page.

<u>Rate limited</u>: 1 per 2 seconds

<u>Response HTTP status code</u>
| Status | Description |
| ------ | ----------- |
| 200 | OK |
| 201 | Created |
| 400 | Bad request |
| 429 | Too many requests |
| 500 | Internal error |

<u>Response body</u>

One of the three
- **newPreLogin**: user has no session yet, one pre-login session is created for them. No body data.

- **existingPreLogin**: user has a pre-login session. No body data.

- **postLogin**: user is authenticated and already has a post-login session, which is returned in this body:

| Name | Data type | Note |
| ---- | --------- | ---- |
| userID | string | |
| userEmail | string | |
| userName | string | |
| signUpChoice | 'GOOGLE' or 'NORMAL' | |

<u>Response header</u>
| Name | Note |
| ---- | ---- |
| X-CSRF-Token| CSRF token generated by server |

### Transaction data endpoints


#### GET /transaction/get

<u>Endpoint</u>: [API host]/transaction/get

<u>Description</u>: Get all transaction records of this user. If the list of transactions is up to date on the client side, it returns 304.
- Clicking refresh will trigger [GET /auth/getCredentials](#get-authgetcredentials), which set a flag that transactions must be fetch.

<u>Rate limited</u>: No

<u>Response HTTP status code</u>
| Status | Description |
| ------ | ----------- |
| 200 | OK |
| 304 | Not modified |
| 400 | Bad request |
| 401 | Unauthorized |
| 500 | Internal error |

<u>Response body</u>

| Name | Data type | Note |
| ---- | --------- | ---- |
| transactionID | string | |
| transactionName | string | |
| accountID | string | |
| value | float | |
| date | string | |
| memo | string | nullable |
